§1 飞书 Agent 心智模型

1.1 产品谱系

飞书在 2025–2026 年形成了三条消费级 Agent 产品线 + 两条开发者路径：

产品定位典型用户进群策略
Aily 智能伙伴 侧边栏个人助手，感知飞书动态 每位员工 1 人 1 个，不进群
Aily 自定义智能体 团队数字员工，管理员搭建 团队 / 部门 一键加入多个群聊
Aily 工作助手 Agentic 循环 + 任务模式 + 虚拟电脑 专业用户 / 全员 侧边栏 + 任务异步
妙搭 OpenClaw 开源 Agent 框架 + 飞书 Channel 极客 / 开发者 自配 bot，可多群
lark-cli + OpenAPI MCP Agent 操作飞书业务域的工具面 Agent / 开发者 不自带 Agent runtime

1.2 Aily 核心实体（OpenAPI）

来源：Aily Session-Run API

实体职责隔离粒度（推断 + 文档）
Session 用户与 Aily 应用的交互会话；含 channel_context、metadata 按 session_id；群聊场景 likely 绑定 chat（待验证 API 创建参数）
Message 会话内消息（文本、表格、图片等） session 内
Run 意图判定 → 技能匹配 → 执行；同 session 仅一个活跃 Run session 内串行

1.3 Channel SDK 能力边界（关键）

来源：接入飞书 Channel SDK

Channel SDK 负责：WebSocket/Webhook、消息解析去重、流式卡片、@ 策略、出站渲染。

Channel SDK 不覆盖：

  • Agent runtime（模型、prompt、工具调用）
  • 多用户隔离：不同 chatId 之间的话题与上下文隔离
  • Session / 上下文持久化
  • 凭据存储

这意味着：飞书把「隔间是否正确」推给 Agent 框架或 Aily 平台层，SDK 本身不兜底。

3.2 Aily 平台对话模式（开发者向）

来源：飞书智能伙伴平台 Aily 的对话模式

  • 模型推理模式：提示词 + Agent 技能，对话只命中该技能
  • 工作流模式：可视化编排，绑定 Workflow 技能
  • 知识问答模式：RAG 优先
  • 混合调度模式：多技能 + 知识问答智能路由

工作流 AI Agent 节点支持记忆子节点（历史对话轮数）与 MCP 工具子节点——记忆粒度由开发者配置，非平台强制隔间。

3.3 关键设计决策（飞书侧）

  1. 交付即用：面向业务人员，弱化底层 Agent 框架暴露。
  2. 飞书生态优先：文档 / 表格 / 任务 / 日历 / 妙记为一等公民工具。
  3. 团队 Agent 复用：自定义智能体强调「100 人共用一个 Agent」+ 记忆隔离，降低重复配置成本。
  4. 无感进化：允许在对话中修改人设 / 技能，加速迭代但扩大 blast radius。
  5. 开发者 escape hatch：Channel SDK + OpenClaw + lark-cli 供极客深度定制。

§4 Claude Tag 隔间模型（精要）

详见仓库笔记：Claude Tag、Agent Identity。

Claude Tag 核心原则：

  • 频道内单一 Claude，所有人可见、可接续。
  • Private channel = 独立 identity；connectors / skills / standing instructions / memory 同边界。
  • Agent 以服务账户行动，非代用户 OAuth。
  • DM 走用户个人 claude.ai，与频道 Agent 双轨分离。

§5 核心分析：「同一身份进多群」

5.1 你的假设 vs 官方说法

层面你的直觉飞书官方判断
同一 bot 进多群 违背 Claude Tag，效果下降 自定义智能体鼓励一键多群 部分成立：问题不在「多群」本身，在隔间粒度
记忆串话 群 A 对话污染群 B 「每个用户、每个群各自一份独立记忆，物理隔离」 隐私层官方有承诺；与 Claude Tag 的 identity 隔间不同层
人设 / 技能 （未明说） 同一 Agent 配置服务所有群 主要风险点：财务群与工程群共享同一 persona / 知识空间
进化 （未明说） 无感发布：对话中修改人设 / 技能 一群进化，全局生效（官方提醒控制开发者扩散）

5.2 三层分离模型

5.3 效果下降的具体机制

  1. Identity 未隔间化：一个「报销专员」Agent 被同时拉进财务群、项目群、客服群——共享同一知识空间与人设，召回时容易引入域外噪声（例如项目术语进入报销回答）。Claude Tag 允许 per-channel override connectors 与 standing instructions；飞书自定义智能体默认无 per-chat 配置分叉。
  2. 无感进化的全局 blast radius：FAQ 第 5 条：无感模式下开发者可在飞书对话中让智能体「学习技能、优化人设」。若未限制开发者范围，一群的对话可改变所有群的 Agent 行为——这比 Claude Tag 的「频道 memory 隔离」更危险，因为改的是共享配置而非仅 memory。
  3. Multiplayer steer 冲突：同一群内多人 @ 同一 Agent，飞书有 Run 串行（Aily API：同 session 仅一个活跃 Run），但跨群并发时同一 Agent 实例可能同时服务异构任务——增加队列延迟与工具调用冲突（推断，待压测验证）。
  4. 自建 Agent 的 session key 风险：OpenClaw 飞书集成中，session key 由 agentId + channel + accountId + chat_id/dmScope 决定。若 binding 将所有群路由到同一 agentId 且未按 chat_id 分 session，则多群共享 LLM 上下文——这是开发者配置问题，Channel SDK 文档已声明不覆盖。
  5. 群历史上下文 buffer：社区方案（如 agent-in-chat-feishu）在 @ 时拉取最近群消息作背景——同一 run 内有效，但若 session 设计错误，群 A 历史可能进入群 B（配置错误场景）。

5.4 你的假设：修正版结论

原表述：「同一身份的助手进入多个不同群聊，违背 Claude Tag 观察，导致效果下降。」

修正版：

  • 对的部分：Claude Tag 的核心是 compartment = 频道级 Agent identity bundle，飞书自定义智能体「一个 Agent 进 N 群」在 Identity 层未实现同等隔间，跨域复用同一 persona / 知识 / MCP 配置会导致专业性下降、召回噪声、进化污染。
  • 需修正的部分：飞书官方在 Memory 层做了 (user × chat) 物理隔离，不等于没有隐私保护；Claude Tag 也不是禁止 bot 物理上存在于多个频道，而是每个频道（尤其 private）有独立 identity 配置。
  • 关键差异：Claude Tag 优化「频道 teammate 一致性 + IAM」；飞书优化「一次搭建、全团队复用 + 飞书工具集成」——产品目标不同，不能直接套用 Tag 规则判「违规」。

6.1 Slack 侧启示

  • slackcli / agent-slack：JSON-first、read-only 模式、结构化 error code——与 lark-cli 同属「Agent-Native CLI」范式。
  • Slack 官方 MCP（2026-02）：OAuth 托管，能力含 search / chat:write / canvas——但没有 Claude Tag 的 compartment IAM；multiplayer 需 Tag 产品层。
  • 对照结论：Slack 生态把「工具面」（CLI/MCP）与「协作 Agent 产品」（Claude Tag）分层；飞书把「Aily 平台」与「lark-cli 工具面」分层，但 Aily 自定义智能体尚未达到 Tag 级 per-channel identity override。

7.3 逼近 Claude Tag 隔间的工程清单

  1. 一群一 Agent 实例：财务 / 工程 / 客服各建独立自定义智能体，不共用知识空间。
  2. 禁用或限制无感进化：生产用标准模式 + 后台发布 + diff 回滚。
  3. Channel SDK 自管 session：session key 必含 chat_id；群聊 never 共享 DM session。
  4. 凭证中心化：沿用 Aily「Agent 不持密钥」；自建 Agent 用 proxy 注凭证。
  5. 审计：Agents.md / 运行日志 + 外部系统 Agent 服务账户归因。
  6. 试点顺序：先在小群 private 试点（对标 Claude Tag「private channel 试点」），再扩 public。
