一句话：Skills 是可渐进披露的文件夹（脚本、模板、按需 Hook、记忆），不是单文件 prompt；Anthropic 内部按九类用途组织，验证类 ROI 最高，description 写给模型当触发检索用。

概览

Claude Code Skills Progressive disclosure Plugin marketplace

Claude Code 团队在生产环境维护数百个活跃 Skills。本文回答：什么类型值得做、如何结构化、何时共享。与 Steering 博文 衔接——彼文讲何时加载，本文讲写什么、怎么写。

与主流做法的差异
  • 社区常把 Skill 当「长 markdown」→ 团队强调 文件夹 + 脚本 + Gotchas
  • 描述写给人看 → 应写触发场景与关键词（启动时仅见 name+description）
  • 知识重复 Claude 默认能力 → 应写组织特有踩坑与反惯性

§1 纠偏：Skill = 文件夹，不是单文件

Claude Code 中 Skills 还可注册动态 Hook。最高效的内部 Skills 同时用好目录结构与配置项。

§2 九类 Skills 分类（标签页）

#类别示例要点
1Library & API referencebilling-lib, sandbox-proxyreference 片段 + gotchas
2Product verificationsignup-flow-driver, tmux-cli-driver内部可度量影响最大；Playwright/tmux + 逐步断言；可录屏

#类别示例要点

5Scaffoldingnew-migration, create-appNL 要求无法纯模板覆盖时用 skill
6Quality & reviewadversarial-review, code-style可挂 Hook 或 GitHub Action
#类别示例要点
7CI/CD & deploybabysit-pr, deploy-<service>可引用其他 skill 取数
8Runbooksoncall-runner, log-correlatorsymptom → tools → 结构化报告

反模式：一个 skill 横跨多类 → 拆分；或上层 skill 按名引用组合（marketplace 尚无原生依赖管理）。

§3 编写心法

原则做法反例
别写废话推模型走出默认惯性（如 design skill 禁 Inter/紫渐变）复述「请写干净代码」
Gotchas从真实失败迭代；append-only 表、跨服务 ID 别名无踩坑记录
渐进披露SKILL.md 列文件树 + 条件指针（pending → stuck-jobs.md）单文件 500 行
别铁轨化给信息与边界，保留适应空间逐步剧本到每个 click
Description触发词：babysit、deploy、oncall给人看的功能简介
脚本库fetch_events() 等 → Claude 写薄编排脚本每轮重写 boilerplate
记忆standups.log、JSON、SQLite；${CLAUDE_PLUGIN_DATA}每次从零
Setupconfig.json；缺配置时 AskUserQuestion硬编码 channel
按需 Hook/careful 拦 destructive bash；/freeze 限编辑目录全局 PreToolUse 常开

Gotchas 示例（原文）

"subscriptions 表 append-only：取最高 version，不是最近 created_at。"
"API gateway 的 @request_id 与 billing 的 trace_id 是同一值。"
"Staging 返回 200 但 Stripe webhook 未处理 → 查 payment_events。"

验证类投资：作者建议可派工程师花一周把 verification skills 做到极好——内部回报高于再加几十行 CLAUDE.md。

§4 分发、组合与度量

  • 小团队：./.claude/skills 入库；每个 skill 略增启动 context
  • 规模化：Plugin marketplace + setup flow，按需安装
  • 组合：skill A 文中引用 skill B 名称 → 已安装则模型 invoke（无原生依赖图）
  • 度量：skill-usage-logging.sh 示例

治理无中央选品——靠使用数据与有机晋升。
